# 项目宪法(constitution)

本文件是**产品与代码层面的不可协商原则**。任何实现决定与它冲突,以它为准;
要推翻其中任何一条,必须用户亲口说。
(协作规则、git 规矩、文件所有权的唯一真相是 `AGENTS.md`,本宪法不重复它;
两者冲突时停下来问用户。)

## 一、内容即文件

每个作品、每篇文章是一个独立 markdown 文件,进 git。
没有数据库,没有后台管理界面。加内容 = 加文件。

## 二、零服务器

构建产物是纯静态网页。任何需要服务器常驻运行的功能(评论、账号、表单后端、
访问统计服务)一概不做,除非用户亲口推翻。

## 三、可迁移性高于一切

换电脑、换 Claude 账号必须无痛(人话步骤见 `docs/搬家清单.md`)。
不引入"只存在于这台电脑"的依赖:本地密钥、全局配置、写死的绝对路径都算违宪。
推到 GitHub 的 git 仓库是唯一权威备份。

## 四、视觉决定集中一处

颜色、字体、间距等视觉变量只住在 `src/styles/tokens.css` 一个文件里,
组件不许私藏。这样"整体改设计方向"只动一处——这是用户明确预告过的场景。

## 五、YAGNI(不为想象中的需求写代码)

功能等真实需要出现了再加。已明确排除的:CMS、评论、账号、会员、
前端框架(React 等)、博客标签、RSS、访问统计。
新增每一个 npm 依赖都要有说得出口的理由。

## 六、双语对等

中文与英文是平等的两个门面,界面文字(导航、按钮、页脚)必须两语齐全。
内容允许单语存在:只有一种语言版本的作品/文章,在另一语言的网站上**直接隐藏**,
不展示、不提示。

## 七、授权合规

任何第三方资产(首当其冲是字体)进入网站发布目录即构成再分发,
必须随附**许可正文本身**。逐个核对许可文件,不类推。
细则见 `AGENTS.md` 设计约束一节(含 RFN/子集化规则)。

## 八、先规格后代码

结构性的改动(新页面、新内容类型、部署方式变更)先改 `docs/specs/` 里的文档、
经用户点头,再动代码。文档与代码不一致时,视为文档过期,修文档。
