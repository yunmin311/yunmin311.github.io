# 素材库板块 — 需求文档(2026-07-17 用户澄清确认)

> 九大板块之七。三轮需求澄清的落字结果,**只定义、不搭建**:
> 页面代码要等视觉方向定稿(`tokens.css`)且用户点名开工才写(宪法第八条,先规格后代码)。
> 内容本身**还没开始有**,文档刻意写轻:定骨架和红线,不预设细节。
> 体例参照 `2026-07-17-photo-gallery-spec.md`。

## 一、定位:它和已定义板块的本质区别

其他板块的条目是「内容/叙事」,看完就完;素材库的条目是「**可复用的物件**」——
像货架/型录,访客是来「拿」或「查」的。而且它是**唯一一个
版权/再分发是第一级属性的板块**:公开仓库+公开网站,素材放上去=向全网再分发,
所以"这东西干不干净"跟"这东西是什么"同等重要。

用户补充的三层(同样是定位的一部分):

- 它也是**自我积累的展示**——不只给访客拿东西,也是"我有这些沉淀"的呈现;
- 它和**用户的另一个私有工具项目有联动**——素材多半是那个项目的提取产物,
  素材库是这些产物的公开出口(流动路径见第四节;该项目与本站完全隔离,
  本文档不写它的名字和细节,只有产物经设计库 public 层流入);
- **信息卡也是可看的内容**——配色/字派这类条目不只是"拿",本身就值得欣赏,
  展示性不比文件型弱。

⚠️ **定位留活口**:用户明说"还有很多没想好"。这份定义是当前共识,不是终稿;
将来用户补充新的定位理解时,回到这份文档更新,别拿现有条文顶回去。

## 二、基本盘(已拍板)

- **内容现状**:还没开始有。板块先定义好,等积累。
- **素材四类**:自制图形素材、自制模板/源文件、收集的公版素材、配色/字派等轻素材。
- **两种形态**(用户确认的二分,页面和字段分别对待):
  - **文件型**(图形/模板/公版收集):有个文件,页面放预览图,是否给下载每条自定;
  - **信息卡型**(配色/字派):内容就是页面上那张卡本身,没有文件可下,
    预览可以由页面自己画(色块、字样),不必出图。
- **展示为主,不一定可下**:是否提供下载是每条自己的开关,不是板块属性。
- **双语**:名称、一句话说明中英各一份。
- **导航名字**:中文站「**素材库**」,英文站「**Assets**」。
- **管理地**(默认约定,沿用影集/播客的理由——字极少,不值得做对接):
  条目直接在网站仓库管,不经过 Obsidian。要改走 Obsidian 得用户亲口说。

## 三、内容模型:一条 = 一个 md 文件

字段(用户确认真会填的,越少越好):

| 字段 | 说明 |
|------|------|
| 名称 + 类型 | 叫什么、属于哪类(图形/模板/公版/配色/字派…类型可扩) |
| 预览图 | 文件型必有,过图片管线;信息卡型可省(页面自己画) |
| 一句话说明 | 这是什么、适合干什么用 |
| 来源/许可 | **收集的公版素材必填**(从哪来、什么许可);自制的可省 |
| 下载(可选) | 开了下载的条目:小文件直接放仓库,大文件挂外链(见下) |

**大文件问题(未决,先记约束)**:模板/源文件(PSD/AI)动辄几十上百 MB,
仓库单文件上限 100MB、整库也就一两个 GB——**大源文件不进仓库**。
放网盘挂外链还是只展示不给文件,等真有大文件了再定,现在不预设。

## 四、素材怎么进网站(两条路都合法,红线只有一条)

- **路一:走设计库**(宪法正道)——私有工具项目的产出 → 用户喊设计中枢收进
  设计库的 `public\` 层 → 网站从 `public\` 复制("只复制不引用")。
  版权关在入库设计库时已经把过。
- **路二:用户手动直接给**——用户把文件直接丢给网站对话。
  这时**版权自查落在入库动作里**:来源哪里、什么许可、能不能公开再分发,
  查清才能进仓库,查不清就问用户,不硬猜。
- **红线(不分哪条路)**:设计库 `private\` 层的东西(他人网站截图、
  付费/禁止再分发材料)**永远不进这个站**;收集类素材没有来源/许可就不入库。
- **字体特别提醒**:字体文件作为素材放出来给人下载,比网页用字更敏感——
  许可正文必须随附、RFN 规则逐个核对(见 AGENTS.md 设计约束),不确定就先问用户。

## 五、页面结构

| 页面 | 内容 | 定稿程度 |
|------|------|----------|
| 素材库列表 | 型录/货架式,按类型能分开看;文件型出预览图,信息卡型页面自绘 | 只定底线 |
| 条目展示 | 预览 + 名称 + 说明 + (收集类)来源许可 + (开了下载的)下载入口 | 结构已定 |

- 具体是网格、分区还是筛选,交给视觉设计,这里只定"按类型能分开看"的底线。
- **提供下载的条目要有授权说明**(允许怎么用):具体措辞等真有可下素材时再定,
  但"给文件必须带说法"这条现在就立住。

## 六、入库工作流(收一件素材的完整动作)

1. **版权检查(第一步,不是最后一步)**:走的哪条路?来源和许可清楚吗?
   `private\` 层的东西直接拒;
2. 预览图过 `scripts/prep-image.mjs`(顺带抹 EXIF/GPS);
3. 建条目 md:名称、类型、一句话说明(中英各一份),收集类填来源/许可;
4. 开下载的:小文件放进仓库;大文件按届时定的方案挂外链;
5. 本地预览确认 → `npm run check` + `npm run build` 双关 → 提交。

## 七、明确不做(YAGNI)

- 不做搜索、标签系统(类型分区够用,量大了再说);
- 不做下载统计、打赏、求购/联系表单;
- 不自建大文件存储(见第三节,约束记下、方案等真需要时定);
- 不做 Obsidian 对接(见第二节,默认约定);
- 不放任何来源/许可不明的东西(这不是功能取舍,是红线,重复一遍以防漏看)。

## 八、开工条件

同时满足才动手:① 视觉方向定稿(`tokens.css` 就位);② 用户点名开工——
内容还没开始积累,这个板块大概率排在有内容的板块之后。
届时实现细节另出实现计划,仍先给用户过目。
