# 个人影集板块 — 需求文档(2026-07-17 用户澄清确认)

> 九大板块之三。本文档是四轮需求澄清的落字结果,**只定义、不搭建**:
> 页面代码要等视觉方向定稿(`tokens.css`)且用户点名开工才写(宪法第八条,先规格后代码)。
> 拿不准的地方回到这份文档;文档没写的,问用户,别套作品集的做法。

## 一、定位:它和作品集的本质区别

**作品集是「每件作品一篇文章」**:有标题、有正文、有叙事,图为文服务。
**影集是「成批的照片本身」**:图就是内容,信息只到相册一级,没有每张图的正文页,
还有排好版的成品直接展示。

所以**不能套作品集模板**——影集没有"正文",数据形状、页面结构、入库动作都不同构。

## 二、内容从哪来、在哪管

- **来源三种**:相机拍的、手机拍的、设计输出的图(视觉稿、渲染图等非拍摄类)。
- **现状**:照片尚未系统管理,散落在相册/文件夹里。影集入库就是管理的起点。
- **管理地(已拍板)**:照片和文字**直接放网站仓库**,不经过 Obsidian。
  理由:影集图多字少(文字只到相册一级、一句话),为它做 Obsidian 双链/附件对接
  是"为很少的文字付很大的对接成本"。Obsidian 对接留给文字为主的板块(博客、学习记录等)。
- **规模**:几百张,按批次增长(一次旅行/一个项目一批)。

## 三、内容模型:以「相册」为单位

信息**填在相册一级**,不逐张填——一批照片基本是同一时间同一地点。

### 相册有两种,不同构,别混为一谈

**① 普通相册**(一批照片 + 一次性填的信息)
- 字段:**标题、时间、地点**(建相册时填一次)。时间地点是仅有的两个信息字段,
  不加说明、标签等更多字段——字段越多入库越累,用户明确只要这两个。
- 相册内照片有顺序(默认按拍摄时间,可手动调)。

**② 现成排版影集**(设计成品,直接展示)
- 形态两种:**排好版的长图**(一张或几张竖长图,照片文字已排进去)、
  **一组按设计顺序的单图**(顺序是设计过的,必须严格按序展示)。
- 页面对它们**只做一件事:按顺序铺出来**,不加字段修饰、不套普通相册的信息栏。
- 默认约定(可推翻):现成影集以整册出现在相册列表,**不拆散进总流**——
  它们是排版成品,不是单张照片。

### 双语(已拍板)

相册的标题、地点**中英各一份**,建相册时一起填。照片本身无语言,两站共用同一批图。

## 四、页面结构

| 页面 | 内容 | 定稿程度 |
|------|------|----------|
| 影集首页 | 相册列表为主 + 一个"按时间看全部"的总流入口 | 结构已定 |
| 相册内页 | 底线:**能看到每张大图**;网格/一列滚动等具体形式交给视觉设计 | 只定底线 |
| 现成影集页 | 按设计顺序铺出长图/单图,别加戏 | 结构已定 |
| 总流 | 普通相册的全部照片按时间排;具体形式交给视觉设计 | 只定底线 |

- 导航名字(已拍板):中文站「**影集**」,英文站「**Photos**」。
- 没有每张照片的独立页面(区别于作品集的详情页)。

## 五、入库工作流(建一个相册的完整动作)

1. 挑照片:从相册/文件夹里选出这一批要发的;
2. 过图片管线:逐张跑 `scripts/prep-image.mjs`(压到合适体积;工序见
   `docs/图片入库工序.md`;HEIC 先转 JPG,工序文档里有办法);
3. 建相册:在影集内容目录建一个相册,放入处理好的照片;
4. 填信息:标题、时间、地点,中英各一份;
5. 本地预览确认 → `npm run check` + `npm run build` 双关 → 提交。

现成排版影集更简单:长图/单图过一遍第 2 步(长图注意别被压糊,必要时提高质量参数),
建册、按序放入、填标题即可。

## 六、隐私与安全红线(仓库和网站都公开)

- 照片入库前**自查**:没有不想公开的人物、场所、生活信息;
- 入库脚本会**抹掉照片的隐藏信息**(EXIF,含 GPS 定位)——原图别绕过脚本直接入库,
  否则拍摄坐标会随仓库公开;
- 上线前照 `docs/上线前检查清单.md` 的隐私段再过一遍。

## 七、明确不做(YAGNI)

- 不做标签、搜索、地图、评论;
- 不做每张照片的详情页/正文页;
- 不做 Obsidian 对接(理由见第二节);
- 不做视频(视频/播客是九大板块里的另一个,等用户点名)。

## 八、开工条件

同时满足才动手:① 视觉方向定稿(`tokens.css` 就位);② 用户点名开工。
届时实现细节(内容集合怎么建、总流怎么生成)另出实现计划,仍先给用户过目。
