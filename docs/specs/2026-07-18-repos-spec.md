# 代码库板块 — 需求文档(2026-07-18 用户澄清确认)

> 九大板块之九,**最后一个**。两轮需求澄清的落字结果,**只定义、不搭建**:
> 页面代码要等视觉方向定稿(`tokens.css`)且用户点名开工才写(宪法第八条,先规格后代码)。
> 体例参照 `2026-07-17-photo-gallery-spec.md`。
> **对接机制不在本文档里**:怎么从 Obsidian 进站已在 `2026-07-18-obsidian-sync-spec.md`
> 定死,本文档只定这个板块**自己**的形状。

## 一、内容来源:混合型——手写介绍 + 仓库链接(先定的岔口)

这个板块澄清的第一刀就是内容从哪来,三条路摆过后用户拍板**混合型**:

- **手写介绍**:每个代码项目在 Obsidian 写一篇(它是什么、为什么做、有什么讲究),
  **走已定的对接管线**,和另两个文字板块同一条路;
- **仓库链接是字段**:活信息(最近提交、代码本身)访客点链接去 GitHub 看,
  网站不替 GitHub 复述——所以**不需要任何新机制**;
- 没选的路记录在案:纯手写型(链接不成字段,页面出不了统一按钮)、
  仓库索引型(信息永远活,但要另建抓取机制,且条目没有自己的话,像目录不像作品)。
  将来若想要"自动抓语言/最近更新",再议专门的小机制,本 spec 不预设。

## 二、定位:名片,以及和三个近邻的分家(全部拍板)

代码库条目 = **项目的名片/入口**:它是什么、用了什么、仓库在哪。一项目一篇、相对静态。

1. **vs 作品集 —— 两头都可以,各讲各的**:作品集讲设计与成果(给雇主/访客),
   代码库讲仓库与实现(给同行)。同一项目(如本网站)两头各一篇、视角不同,
   可选互挂(用项目记录那套关联字段的思路);
2. **vs 项目记录 —— 名片 vs 过程**:代码库一篇静态名片;项目记录一串持续生长的
   过程与反思。可选互挂;
3. **vs 学习记录 —— 练习也收,标记为练习**(用户拍板,和我推荐的"只收拿得出手的"
   不同,以用户为准):跟教程做的练手仓库也上架,但**类型标记必须是"练习"**、
   页面上和正式项目分区(见第五节)——全量真实展示,同时第一印象不被稀释。
   练习过程的心得照旧归学习记录,那边正文可以链过来。

## 三、内容模型:一篇 = 一个 md 文件(名片)

字段(基线三件套之外,本板块特有的):

| 字段 | 说明 |
|------|------|
| 标题 + 日期 + 一句话简介 | 基线三件套(对接 spec 第四节,缺一样同步时拦下) |
| **仓库链接** | **必填**——混合型的根基,页面"去看代码"按钮从它来 |
| 类型标记 | 正式 / 练习,二选一(分区靠它) |
| 技术栈 | 几个短词(如 Astro、TypeScript),列表页一眼扫出这人会什么;手写维护,不自动抓 |

- 双语照旧可选(对接 spec 已定):默认中文,想双语的在 `en/` 写同名英文版;
- **导航名字**:中文站「**代码库**」,英文站「**Repos**」。

## 四、红线:上架 = 给仓库导流,先自查

1. **列出的仓库必须公开可访问**:私有仓库的链接访客点了吃闭门羹,
   而且把私有仓库的名字/存在公开本身就是泄露——没公开的仓库不上架;
2. **客户/公司的代码仓库不列**(不论权限如何),那个私有工具项目的仓库也不列
   (私有,且照既定脱敏规矩连名字都不进公开文档);
3. **上架前给仓库做一次体检**:README、代码、提交历史里没有密钥、
   个人敏感信息、内部信息——仓库在 GitHub 公开是那边的动作,
   但列进网站等于主动给它导流,体检责任落在上架这个动作里;
4. 介绍配图(截图等)照旧走管线(对接 spec 的自动关卡),画面内容自己把关。

## 五、页面结构:一个列表、两个区

| 页面 | 内容 | 定稿程度 |
|------|------|----------|
| 板块首页 | 单列表两区:**正式项目打头阵,练习区在下方标清楚**;每条出标题+简介+技术栈 | 只定底线 |
| 条目页 | 手写介绍正文 + 醒目的"去看代码"按钮(仓库链接);有互挂的出关联链接 | 结构已定 |

- 不需要归组机制(一项目一篇,量不大),不按技术栈分组(项目多了再议);
- 具体版式交给视觉设计,这里只定"正式/练习分区、每条能跳仓库"的底线。

## 六、写作与发布流(上架一个项目的完整动作)

1. 先过第四节体检:仓库公开了吗?里面干净吗?客户/私有的直接不上;
2. 在 Obsidian 该板块目录写介绍(名片,不用长);
3. 填字段:基线三件套 + 仓库链接 + 类型标记(+ 技术栈);
4. 移进发布文件夹 → 私有笔记库按同步按钮(对接 spec 管线接手);
5. 网站要更新时照旧手动部署(同步 ≠ 上线)。

## 七、明确不做(YAGNI)

- 不自动抓仓库元数据(语言、星数、最近更新)——访客点链接看活的,想要再议;
- 不做状态字段(活跃/归档)——最近提交在 GitHub 上一目了然,字段只会陈旧;
- 不做按技术栈分组、筛选、搜索;两区一列表够用;
- 不内嵌代码浏览、README 渲染——不替 GitHub 干活;
- 不做星数/下载量等任何统计展示。

## 八、开工条件

同时满足才动手:① 视觉方向定稿(`tokens.css` 就位);② 对接管线已按
`2026-07-18-obsidian-sync-spec.md` 建成跑通;③ 用户点名开工。
届时实现细节另出实现计划,仍先给用户过目。
