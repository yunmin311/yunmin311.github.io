# 项目记录与反思板块 — 需求文档(2026-07-18 用户澄清确认)

> 九大板块之五,三个文字类板块里第一个被点名定义的。两轮需求澄清的落字结果,
> **只定义、不搭建**:页面代码要等视觉方向定稿(`tokens.css`)且用户点名开工才写
> (宪法第八条,先规格后代码)。体例参照 `2026-07-17-photo-gallery-spec.md`。
> **对接机制不在本文档里**:怎么从 Obsidian 进站(私有侧推、出门时翻双链、
> 图片自动过关、字段基线三件套)已在 `2026-07-18-obsidian-sync-spec.md` 定死,
> 本文档只定这个板块**自己**的形状,不重开地基。

## 一、定位:它和作品集的本质区别

同一个项目可能两头都出现,但分工完全不同:

- **作品集是门面**——放成品,给访客/雇主看,一件作品一篇文章;
- **本板块是后厨**——放过程和反思,给同行和自己看:项目怎么做的、
  中间怎么想的、事后回头看学到了什么。

**条目两种形状都要**(用户拍板),这正是"记录**与**反思"两个词:

1. **过程日志型**:项目进行中随时记,一个项目累积多篇,碎而勤;
2. **复盘型**:项目收尾后写一篇整体反思,一个项目一篇,少而重。

所以一个项目在这里是**一串文章**,不是一篇——归组机制因此是必需品(见第三节)。

## 二、基本盘(已拍板)

- **写作地**:Obsidian 发布文件夹的本板块子目录(结构见对接 spec 第二节),
  这是三个文字类板块的共同前提;
- **项目范围**:自己的个人项目、客户/公司项目、那个私有工具项目**都可以写**
  ——各自的红线见第四节;
- **双语**:像设计整合那样可选(对接 spec 已定)——默认中文,
  想双语的篇目在 `en/` 写同名英文版;
- **导航名字**:中文站「**项目记录**」,英文站「**Projects**」
  (和作品集 Works 分开:一个是成品门面、一个是过程后厨);
- **与作品集可选挂钩**:有对应作品的反思,填一个选填字段指向那件作品,
  反思页出"去看成品"的链接;没有对应作品的项目(学习项目、失败项目)不填就是。

## 三、内容模型:一篇 = 一个 md 文件,项目靠字段归组

字段(基线三件套之外,本板块特有的):

| 字段 | 说明 |
|------|------|
| 标题 + 日期 + 一句话简介 | 基线三件套(对接 spec 第四节,缺一样同步时拦下) |
| **项目名** | **必填,归组键**——同一项目的所有篇目填同一个名字,页面靠它聚合 |
| 类型标记 | 过程记录 / 复盘,二选一(项目页里复盘要认得出来) |
| 关联作品 | 选填,填作品集里对应作品的文件名;填了就出互跳链接 |

- **项目本身不需要单独建档案文件**:项目页由归组键自动聚合该项目的所有篇目,
  不另写"项目简介文件"——复盘篇天然承担"这个项目是什么"的角色;
- 项目名一旦定了别改:它是归组键,改名等于把一串文章拆成两组
  (真要改,同项目所有篇目一起改,一次同步过去)。

## 四、红线:反思文字比作品展示更容易漏东西

反思写的是"怎么想的、怎么做的",最容易带出不该公开的内容。三条按项目类型:

1. **「隐藏、只有我能看到」= 不放进发布文件夹**(用户确认)。
   技术现实先摆诚实:公开仓库+静态站,**不存在"在线但只有我可见"**——
   页面不列出也没用,文件本身躺在公开仓库里谁都翻得到。
   真正的私密反思写在发布文件夹**外**,只活在 Obsidian(私有库)里,
   一个字都不出门;哪天想公开了,移进发布文件夹即可。
   发布文件夹就是那个开关,不另造机制;
2. **客户/公司项目**:客户名、报价、内部流程、保密细节、未公开功能
   **一律不写**;项目脱敏到"看不出是哪家"才能发,拿不准的整篇留在文件夹外;
3. **那个私有工具项目**:照素材库/设计整合的既定规矩脱敏——
   不点名、不写定位细节(本仓库公开,私有项目身份不进公开文档)。

配图照旧走管线(对接 spec 的自动关卡,EXIF/GPS 出不了门)——
但注意**截图内容本身**:带客户信息的界面、内部工具的界面,抹的是元数据不是画面,
画面里的东西自己把关。

## 五、页面结构:项目为纲 + 全站时间流入口

| 页面 | 内容 | 定稿程度 |
|------|------|----------|
| 板块首页 | 项目列表(每项目一张卡:项目名 + 篇数 + 最近更新) | 只定底线 |
| 项目页 | 该项目全部篇目按时间排;复盘篇标出/置顶 | 只定底线 |
| 文章页 | 普通文章版式,和全站统一;有关联作品的出互跳链接 | 结构已定 |
| 时间流入口 | 全板块所有篇目按时间倒序的"最近更新"视图 | 只定底线 |

- 具体版式交给视觉设计,这里只定"按项目能聚合、复盘认得出、有一个时间流视图"的底线;
- 时间流是**入口/视图**,不是另一套内容——同一批文章的两种看法。

## 六、写作与发布流(发一篇的完整动作)

1. 在 Obsidian 该板块目录写(过程记录随时记;复盘收尾写);
   私密的写在发布文件夹外;
2. 填字段:基线三件套 + 项目名(+ 类型标记;有对应作品填关联作品);
3. 发布前自查第四节红线(客户信息、私有项目点名、截图画面);
4. 移进发布文件夹 → 去私有笔记库按同步按钮(对接 spec 第三节的管线接手:
   翻双链、图片过关、验票、落进公开仓库);
5. 网站要更新时照旧手动部署(同步 ≠ 上线)。

## 七、明确不做(YAGNI)

- 不做"在线但仅自己可见"的加密/隐藏页(见第四节,发布文件夹就是开关);
- 不做项目状态标记(进行中/已完结)——项目页看最近更新日期就知道活不活;
- 不做标签、搜索、评论;项目归组够用;
- 不做进度条、时间线图表这类装饰性结构;
- 不写"项目档案文件"(见第三节,复盘篇承担这个角色)。

## 八、开工条件

同时满足才动手:① 视觉方向定稿(`tokens.css` 就位);② 对接管线已按
`2026-07-18-obsidian-sync-spec.md` 建成跑通(本板块是它的第一个真实用户);
③ 用户点名开工。届时实现细节另出实现计划,仍先给用户过目。
