# Obsidian → 网站 对接 — 横切需求文档(2026-07-18 用户澄清确认)

> 这不是某一个板块,是**三个文字类板块(项目记录与反思 / 学习记录 / 代码库)共同的
> 前置骨头**:大段文字在 Obsidian 里写,怎么安全地变成网站内容。
> **只定义、不搭建**:本轮不写代码、不装插件、不改配置、不建任何钥匙/secret
> (宪法第八条,先规格后代码)。体例参照 `2026-07-17-photo-gallery-spec.md`。
> 文中「已查证」的结论,均为 2026-07-18 对 npm/GitHub 当下状态的实查,不是印象。

## 一、这是什么、为什么现在定

愿景文档早就点了名:内容源是 Obsidian(跨项目底座),而 Obsidian 的 md ≠ 网站的 md,
「对接必须专门处理,不是配一下就好」。影集/播客/素材库当初选"直接在仓库管"是因为字少;
三个文字类板块是大段文字,**必须在 Obsidian 里写、再进站**——这份文档定的就是
"再进站"这条路。用户的笔记整库已同步在一个**私有 GitHub 仓库**里。

## 二、三大方向(已拍板)

### 1. 搬运走「私有侧推」(方向 B)

私有笔记库自己跑一个自动动作(GitHub Actions),把发布文件夹的内容处理好后
**推进公开网站仓库**;公开仓库**不持有任何能碰笔记库的钥匙**。

为什么不是反过来(公开侧拉)——查证到的关键事实:

- **GitHub 的读权限最小粒度是「整个仓库」**,不存在"只许读某个文件夹"的钥匙
  (deploy key / fine-grained PAT / GitHub App 三种全查过,都一样)。
  所以"拉"的方案里,公开仓库手里那把钥匙**事实上能读整个笔记库**,
  "只拿发布文件夹"只是代码里的自觉,不是权限上的墙;
- 钥匙泄露的损失不对等:"拉"的钥匙泄露 = 全部笔记(日记、草稿)一次性外泄,
  **看过就收不回**;"推"的钥匙只能写公开网站仓库,泄露了大不了内容被改,
  撤销 + 换钥匙即恢复;
- "推"的结构里,**决定"什么出门"的闸完全握在私有侧**——权限结构和意图对齐。

实现形态(已查证):现成的跨仓库推送工具都已停止维护,主流做法是**十几行自写
git 命令**(官方 checkout + 复制 + 推送),不引第三方,凭据不过外人手。
钥匙将来建的时候用**只限公开仓库、只有写内容权限**的细粒度 PAT(个人账户可不设到期)
或公开仓库的一把写入 deploy key,存进**私有笔记库**的 secrets——本轮不建,
届时由用户在 GitHub 页面亲手建(给人话步骤;照纪律,永不让用户把钥匙交给对话)。

### 2. 选择靠「文件夹」,不靠标记

vault 里开一个**专门的发布文件夹**,只有它会被同步——一眼看清什么会公开。
文件夹内部结构照网站现有惯例的镜子:

```
发布文件夹/(名字由用户在 vault 里自定)
├── reflections/   项目记录与反思
│   ├── zh/xxx.md
│   └── en/xxx.md(可选,写了才有)
├── learning/      学习记录
│   └── zh/…  en/…
└── code/          代码库
    └── zh/…  en/…
```

- **放进文件夹 = 要发布,移出去 = 撤下**——所以不需要 draft 草稿字段,
  没写完的别放进来就是了;
- **永不同步整库**,这是红线(见第六节),不是配置项。

### 3. 方言翻译在「出门时」做,不在站内做

Obsidian 的 `[[双链]]` 在出门那一刻就翻成标准 markdown,公开仓库收到的是成品;
网站构建保持普通,不加双链插件。这么选的隐私理由(调查后才浮出来的):
若原样 `[[ ]]` 进公开仓库、靠站内构建时的插件转,那么「[[某篇私密笔记标题]]」
会**原文躺在公开仓库源码里**——页面上看不见,但任何人翻仓库文件都看得见。
出门时翻掉,`[[ ]]` 语法和私密笔记的痕迹根本不进公开仓库。

工具(已查证,推荐方向):**obsidian-export**——维护健康(2025-03 仍在发版)、
原生支持只导出一个子文件夹、链到未导出笔记的双链**自动退化成纯文字**,
三条硬需求直接命中。两处行为文档没写透,开工时要先实测(见第八节)。
备胎也已核实:`@flowershow/remark-wiki-link` 4.0(2026-06 刚发版,活跃维护)
+ 约十行的自写降级小插件,走"站内构建时转"的路——实测不合意就换它。

## 三、管线全景:按一下按钮之后发生什么

**触发:手动按钮为主**(去私有笔记库点一下才跑)——多一道人的确认,
和"暂不上线、先存档"的节奏一致;将来想自动是改一行配置的事。
注意:同步进公开仓库 = **全网可见**(仓库本身公开),不等于网站上线
(网站部署照旧另一个手动按钮,两不相干)。

私有笔记库的动作,按顺序:

1. **圈地**:只取发布文件夹(obsidian-export 按子文件夹导出);
2. **翻译**:`[[双链]]` → 普通链接;链到未发布笔记的 → 退化成纯文字;
   `![[图片嵌入]]` → 标准图片引用;
3. **图片关卡(红线,自动过)**:附件逐张跑 `scripts/prep-image.mjs`
   (脚本从公开仓库取——动作本来就要检出公开仓库才能推送,顺带就有了),
   压缩 + 抹 EXIF/GPS 后才放行;HEIC 处理不了就当场报错拦下,
   先照 `docs/图片入库工序.md` 的办法转成 JPG;
4. **验票**:frontmatter 基线字段(见第四节)缺一样就拦下报错,不让半残上站;
5. **落位**:改写图片路径成网站惯例,把三个板块各自的 md + 图
   放进公开仓库 `src/content/` 下对应目录,提交、推送。

胶水层的体量(已查证有社区先例佐证):**百行以内的 Node 脚本**,不是一个工程。

## 四、网站侧:三个板块各自一个内容集合

- **各自一个 collection**(用户拍板)——愿景警告过"别把内容模型焊死成一个模子",
  三个板块将来各自扩字段(代码库挂仓库链接、学习记录挂课程来源)互不牵连;
  同步管线是同一条,只是终点分三处;
- **frontmatter 基线(三板块共同最低配置)**:**标题 + 日期 + 一句话简介**,
  和博客同款三件套。日期**必填且手写**——从文件修改时间猜日期在 git 同步里
  不可靠(每次同步都可能刷新时间)。板块专属字段等各板块自己的需求澄清时再加;
- **同步落点目录是机器管理区**:`src/content/` 下这三个板块的目录由管线写入,
  **人不手改**——改了会被下一次同步覆盖。想改内容,回 Obsidian 改了重新同步。

## 五、双语:像设计整合那样可选

默认写中文;哪篇想双语,就在该板块的 `en/` 下写同名英文版,同步后双语互切;
没写英文的只在中文站出现。三个板块都是大段文字,不让翻译工作量拖着不敢发。

## 六、红线(方案必须同时满足,违反即作废)

1. **附件必过 `scripts/prep-image.mjs`**(抹 EXIF/GPS)。关卡设在同步管线里
   自动执行——靠机器不靠记性;**不开任何"vault 原图直接拷进仓库"的旁路**,
   否则拍摄坐标随公开站泄露;
2. **只发布那个专门文件夹,永不同步整库**。链到未发布笔记的双链一律降级成纯文字,
   不渲染链接、不暴露目标。**一个诚实的提醒**:降级后留下的"纯文字"仍是你
   写在正文里的那几个字——机器能保证它不成为链接、`[[ ]]` 语法不进公开仓库,
   但不能替你把字变没。真正私密的笔记名,别写进要发布的正文里;
3. **笔记 vault 独立、干净、只装笔记**,别和别的私有项目混一个仓库。
   选了"推"之后公开侧本来就摸不到笔记库,这条的压力小了,但仍然成立:
   同步动作跑在笔记库里,库越干净,动作能碰到的东西越少;
4. **同步动作里不引第三方组件经手凭据、不打印凭据**——凭据只交给 GitHub
   官方的检出步骤(这也是不用那些弃维护的现成推送工具的又一理由)。

## 七、明确不做(YAGNI)

- **不装任何 Obsidian 插件**。查证过最活跃的发布类插件(Enveloppe):
  要用户自己配 PAT、发布动作在 Obsidian 应用内——和"永不向用户要钥匙"
  "只有网站对话跑 git"两条纪律正面相抵,明确排除;
- 不支持 `![[笔记嵌入]]`(整篇笔记嵌进另一篇)——只做图片嵌入和双链跳转;
- 不做整库同步、不做站内 graph view / 反向链接 / 搜索;
- 不做"同步完自动部署上线"——两个按钮就是两道闸,故意的;
- 不迁就 vault 里其他 frontmatter 习惯(标签、别名等一概不读),只认基线三件套。

## 八、开工条件与开工第一批动作

同时满足才动手:① 视觉方向定稿(`tokens.css` 就位);② **至少一个文字类板块
完成了自己的需求澄清**(三个板块要用用户的专门启动方式,本文档不越位替它们定义);
③ 用户点名开工。

届时的第一批动作(顺序写死,免得到时踩坑):

1. **实测 obsidian-export 两处欠文档化行为**:链到"vault 里根本不存在的文件"
   的双链怎么处理;附件复制出来的目录结构长什么样。不合意就换备胎
   (@flowershow 插件路线,或纯 Node 自写、约两三百行);
2. 用户在 GitHub 页面亲手建钥匙(给人话步骤),存进私有笔记库的 secrets;
3. 先拿一两篇真笔记跑通整条管线、人工核对公开仓库收到的东西,再谈批量。
   实现细节届时另出实现计划,仍先给用户过目。
